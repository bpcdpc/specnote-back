## 액터(Actor) 정의

| 액터 | 설명 |
| --- | --- |
| 백엔드 개발자 | NestJS 프로젝트를 개발하고 API를 설계. SpecNote에서 프로젝트를 생성·운영하는 경우가 많음 |
| 프론트엔드 개발자 | API 스펙을 확인하고 엔드포인트 단위로 논의 |
| 비개발 직군 (기획/디자인/QA) | 엔드포인트 댓글을 통해 의견 제시 |
| AI Agent | 댓글 스레드 요약역할을 수행하는 역할 |

### 역할(Role) — 프로젝트 스코프

역할은 계정 전역이 아니라 프로젝트마다 상대적으로 부여된다. 위 3개 액터 누구나 어떤 프로젝트에서는 Owner, 다른 프로젝트에서는 Member가 될 수 있다.

| 역할 | 부여 방식 | 권한 요약 |
| --- | --- | --- |
| Owner | 프로젝트 생성자 | 프로젝트 생성, 수정, 삭제(+ Member 권한 전부) / 스펙 등록·업데이트, 멤버 초대·제외 |
| Member | Owner의 초대 | 엔드포인트 조회, 댓글/대댓글 작성·수정·삭제(본인 것), 멘션, 리액션 |

> User : 서비스에 가입한 계정
AI Agent : 미리 정의된 AI용 계정
Owner : 프로젝트를 만든 사람
Member : 프로젝트에 초대된 사람
>

## 유스케이스 목록

### UC-0. 가입 및 로그인

| 항목 | 내용 |
| --- | --- |
| 액터 | 전체 사용자 (Owner/Member 되기 이전의 User) |
| 목표 | 이메일 기반으로 계정을 만들고 로그인하여 SpecNote에 접근한다 |
| 사전조건 | 없음 (누구나 가입 가능) |
| 기본 흐름 | 1. 가입 화면에서 이름/이메일/비밀번호 입력 2. 시스템이 비밀번호를 해시(bcrypt 등)하여 계정 생성 3. 이메일/비밀번호로 로그인 → 시스템이 JWT 발급 4. 로그인 상태에서만 프로젝트/엔드포인트/댓글 화면에 접근 가능 |
| 예외 흐름 | 이미 가입된 이메일로 가입 시도 → 가입 실패 |
| 비고 | 이메일 인증·비밀번호 찾기·전역 관리자 권한은 본 문서 범위에서 다루지 않음. 접근 통제는 계정 권한이 아니라 **프로젝트 멤버십**이 담당 |
| 사용 예시 | 희경 님이 이름/이메일/비밀번호만으로 가입 후 로그인 → 이후 초대받은 프로젝트에 접근 가능 |

### UC-1. 프로젝트 생성 및 스펙 등록

| 항목 | 내용 |
| --- | --- |
| 액터 | Owner (프로젝트를 생성하는 순간 해당 프로젝트의 Owner가 됨) |
| 목표 | 새 프로젝트를 만들고 OpenAPI 스펙을 등록한다 |
| 사전조건 | 로그인 상태 |
| 기본 흐름 | 1. OpenAPI 스펙을 받아올 주소(URL)를 입력해 프로젝트 생성 2. 생성자가 자동으로 Owner 역할을 부여받음 3. 시스템이 해당 주소에서 스펙을 가져와 파싱하고, `title`/`description`/`version`/`oasVersion`을 스펙의 `info.*`·`openapi` 필드에서 추출해 프로젝트 정보로 저장(사용자 직접 입력 없음, `description` 외 필수) 4. 파싱한 엔드포인트를 해당 프로젝트 스코프로 DB에 저장 5. 원본 스펙 JSON 전체를 첫 번째 버전 스냅샷으로 저장 6. Try it out에 사용할 API 서버 base URL을 입력(비필수) |
| 예외 흐름 | 입력 내용이 OpenAPI 형식이 아니거나 주소 접근·파싱에 실패 → 에러 메시지 표시, 저장 취소 |
| 비고 | 스펙 입력 수단은 외부 URL fetch. **프로젝트 `title`/`description`/`version`/`oasVersion`은 사용자가 직접 입력·수정하지 않고 스펙 파싱값으로만 채워지며, 스펙 리로드(UC-10) 시에만 최신값으로 갱신된다.** 저장된 스냅샷은 append-only로 쌓인다. base URL은 Try it out(UC-4) 전용 선택 항목 — 스펙 JSON 소스와는 별개이며, 미입력 시 Try it out만 비활성될 뿐 스펙 열람·댓글 등 나머지 기능은 영향 없음 |
| 사용 예시 | 백엔드 개발자가 개발서버 스웨거의 `/api-json` 주소를 입력해 프로젝트를 생성 → 스펙의 `info.title`이 "수강신청 시스템"으로 파싱되어 프로젝트명이 되고, `Courses`, `Users`, `Auth`, `Enrollments` 엔드포인트가 파싱·저장되며 원본 JSON이 v1 스냅샷으로 보관됨. 이어서 Try it out을 쓰기 위해 개발 서버 주소 `https://dev.example.com/api`을 base URL로 입력(생략 가능) |

### UC-2. 멤버 초대 및 제외

| 항목 | 내용 |
| --- | --- |
| 액터 | Owner |
| 목표 | 초대된 팀원만 프로젝트의 스펙과 댓글에 접근하도록 멤버를 관리한다 |
| 사전조건 | 초대 대상이 이미 SpecNote에 가입되어 있어야 함 |
| 기본 흐름 | 1. Owner가 이메일 완전일치로 기존 가입자를 검색 2. 검색된 사용자를 Member로 초대 3. 시스템이 초대 대상에게 서비스 내부 알림을 발송 4. 초대된 사용자는 해당 프로젝트에 접근 가능 |
| 대안 흐름 | Owner가 멤버를 프로젝트에서 제외 → 해당 사용자는 즉시 접근 차단 |
| 예외 흐름 | 이메일이 가입자와 완전일치하지 않으면 초대 대상 없음으로 처리 |
| 비고 | 이름은 검색 키가 아니라 표시용(전체 유저 명단 노출 방지). 멤버십은 물리 삭제하지 않고 `isRemoved` 단일 boolean 소프트 딜리트 마커로 제외 여부만 표시한다(별도 이력 로그는 두지 않음). 알림은 내부 알림으로만 전달(메일 발송은 다루지 않음) |
| 사용 예시 | Owner인 백엔드 개발자가 프론트엔드 개발자와 QA의 이메일을 각각 입력해 초대 → 두 사람에게 내부 알림이 도착하고 프로젝트에 접속 가능. 이후 한 명을 제외하면 접근은 차단되고, 해당 멤버십은 `isRemoved` 마커로 "제외됨" 표시가 됨 |

### UC-3. 프로젝트 선택 및 스펙 탐색

| 항목 | 내용 |
| --- | --- |
| 액터 | Owner, Member |
| 목표 | 접근 권한이 있는 프로젝트를 골라 진입하고, 사이드바에서 엔드포인트를 탐색한다 |
| 사전조건 | 해당 프로젝트의 Owner이거나 초대된 Member이며, 스펙 파싱이 완료됨 |
| 기본 흐름 | 1. 내가 생성했거나 초대받은 프로젝트 목록 중 하나를 선택해 진입 2. 사이드바에 태그별로 그룹핑된 엔드포인트 목록 표시 3. 엔드포인트 클릭 시 우측 디테일 패널 갱신 |
| 대안 흐름 | 검색어 입력 시 엔드포인트 목록을 실시간 필터링 |
| 비고 | 모든 조회는 **프로젝트 스코프**로 제한. 프로젝트 목록에는 본인이 Owner인 프로젝트와 Member로 초대된 프로젝트만 노출됨 |
| 사용 예시 | 프론트엔드 개발자가 초대받은 프로젝트 목록에서 "수강신청 시스템"을 선택해 진입 → `Courses`, `Enrollments` 등 태그 아래 엔드포인트가 정리되어 표시됨 |

### UC-4. 엔드포인트 상세 확인 및 요청 테스트

| 항목 | 내용 |
| --- | --- |
| 액터 | Owner, Member |
| 목표 | 특정 엔드포인트의 요청/응답 스키마와 example을 확인하고, 필요 시 실제 요청을 보내 동작을 테스트한다 |
| 사전조건 | 프로젝트에 진입한 상태 |
| 기본 흐름 | 1. 사이드바에서 엔드포인트 클릭 2. 우측 디테일 패널에 요청 스키마(필드명·타입), 응답 스키마, example 값 표시 |
| 대안 흐름 | 1. 배열 안에 중첩된 객체 구조(예: `timeTables`)를 재귀적으로 펼쳐 확인 2. **"Try it out"으로 요청 파라미터/바디를 example 값 기반으로 입력(수정 가능) → 프로젝트에 등록된 base URL로 실제 요청 전송 → 실제 응답(상태 코드·바디)을 화면에 표시.** 3. 인증이 필요한 엔드포인트의 경우 전역 Bearer 토큰을 입력해 요청 헤더에 포함(http bearer 타입만 지원) |
| 예외 흐름 | 1. 프로젝트에 base URL이 입력되지 않았으면 Try it out은 **비활성**(스펙 열람·댓글 등 나머지 기능은 정상) 2. 대상 API가 SpecNote 오리진의 요청을 CORS로 허용하지 않으면 요청이 실패하며, 원인이 SpecNote가 아닌 대상 API의 CORS 설정에 있음을 안내 3. 엔드포인트가 http bearer 외 인증 방식(oauth2 등)을 요구하면, 해당 인증 타입은 지원하지 않음을 안내 |
| 비고 | 요청 대상 base URL은 **UC-1(프로젝트 생성) 시 Owner가 입력한 값**(비필수). 이후 프로젝트 설정에서 Owner가 수정 가능(스펙 등록·업데이트와 동일한 권한선상). Try it out은 실제 API 서버에 직접 요청을 보내므로 목 서버(가짜 응답)와 달리 **실제 시스템의 데이터를 변경할 수 있음** — 이는 이 기능의 성격이며 SpecNote가 통제하는 영역이 아님. 인증은 전역 Bearer 토큰 입력을 지원하되 **http bearer 타입만** 대상으로 하며, oauth2 등 그 외 인증 타입은 미지원으로 안내한다. base URL과 CORS는 스펙 JSON 소스(업로드/복붙)와 무관하며, 서로 영향을 주지 않음 |
| 사용 예시 | 프론트엔드 개발자가 `POST /courses`를 클릭 → `title`(string), `courseType`(enum), `credits`(number), `timeTables`(array, `dayOfWeek`/`hour`) 등 필드와 example을 확인해 입력 폼에 반영. 이어서 Try it out으로 example 값을 그대로(또는 일부 수정해) 전송 → 실제 응답으로 `201 Created`와 생성된 리소스를 확인. 인증이 필요한 요청에는 전역 Bearer 토큰을 입력해 헤더에 실어 전송 |

### UC-5. 엔드포인트 댓글 작성 / 수정 / 삭제

| 항목 | 내용 |
| --- | --- |
| 액터 | Owner, Member |
| 목표 | 특정 엔드포인트에 대해 설계 의견이나 질문을 최상위 댓글로 남긴다 |
| 사전조건 | 프로젝트에 진입해 대상 엔드포인트를 선택한 상태 |
| 기본 흐름 | 1. 엔드포인트 선택 2. 댓글 입력폼에 작성 3. 댓글 스레드 최상위(1뎁스)에 등록되며 작성자/작성 시간과 함께 표시 |
| 대안 흐름 | 1. 본인이 작성한 댓글 수정 2. 본인이 작성한 댓글 삭제 → 내용만 "삭제된 댓글입니다"로 대체되고 작성자/시간/하위 대댓글 등 구조는 유지(소프트 딜리트) |
| 비고 | 여기서 다루는 것은 최상위 댓글(1뎁스)의 작성/수정/삭제만이다. 기존 댓글에 이어 다는 대댓글 작성은 **UC-6**을 참고. **이슈 제목/본문/상태 같은 개념은 두지 않음** — 순수 텍스트 댓글 구조만 유지 |
| 사용 예시 | 프론트엔드 개발자가 `POST /enrollments`에 "정원이 꽉 찼을 때 에러 코드가 명세에 없는데 어떻게 처리되나요?"라고 댓글 작성 |

### UC-6. 댓글에 대댓글 작성 / 수정 / 삭제

| 항목 | 내용 |
| --- | --- |
| 액터 | Owner, Member |

### UC-10. 스펙 업데이트

| 항목 | 내용 |
| --- | --- |
| 액터 | Owner |
| 목표 | 사용자의 백엔드 API가 바뀌었을 때 프로젝트의 엔드포인트 목록을 최신 상태로 갱신한다 |
| 사전조건 | 프로젝트의 Owner이며, 갱신할 스펙 내용을 확보한 상태 |
| 기본 흐름 | 1. Owner가 새 스펙을 커밋 (버튼 클릭) 2. 프로젝트 설정 시 연결한 URL에서 시스템이 새 json을 얻어와 파싱 후 기존 저장분과 비교 3. 새 엔드포인트 추가 / 변경된 필드(summary 등) 갱신 / 사라진 엔드포인트는 소프트 삭제(삭제 표시) / **이전에 소프트 삭제됐던 엔드포인트가 같은 `path+method`로 재등장하면 새 행을 만들지 않고 기존 행을 부활(`isDeleted=false`)시키고 필드를 갱신** 4. 새로 받아온 원본 JSON을 **새 버전 스냅샷으로 append** 5. 커밋 이후 발생하는 요청/새로고침부터 최신 스냅샷이 적용되며, 이미 열람 중인 클라이언트는 스냅샷 버전 불일치를 감지하면 "스펙 업데이트됨" 배너를 표시하고 사용자가 새로고침할 때 최신 상태로 반영 |
| 비고 | 동일 엔드포인트 판단 기준은 `path + method` 조합. 갱신 권한은 Owner에 한정. 시스템은 엔드포인트 상세 응답에 스냅샷 버전 식별자를 포함하고, 클라이언트가 캐시한 `components` 버전과 불일치하면 조용한 자동 교체 대신 배너로 알린 뒤 **사용자 주도 새로고침**으로 최신 스냅샷을 반영한다(진행 중이던 댓글 맥락 보호). 버전은 덮어쓰지 않고 쌓이며(append-only), 열람 기본 대상은 항상 **최신 스냅샷** (과거 버전 열람·비교 UI는 본 문서 범위 밖). `path+method`가 동일한 엔드포인트가 삭제 후 재등장하면 기존 소프트삭제 행을 부활시켜 댓글 스레드 연속성을 유지한다 |
| 사용 예시 | 백엔드 개발자가 `PATCH /enrollments/:id`를 추가하고 `DELETE /courses/:id`를 제거한 새 스펙으로 업데이트 → 사이드바에 `PATCH /enrollments/:id`가 추가되고 `DELETE /courses/:id`는 "삭제됨" 표시로 전환. 열람 중이던 팀원 화면에는 "스펙 업데이트됨" 배너가 뜨고, 새로고침하면 최신 목록이 반영됨. 이전 버전 스냅샷은 그대로 보존 |

### UC-12. 댓글 스레드 이동

| 항목 | 내용 |
| --- | --- |
| 액터 | Owner |
| 목표 | 한 엔드포인트에 쌓인 논의(댓글 스레드)를 관련 있는 다른 엔드포인트로 옮긴다 |
| 사전조건 | 프로젝트의 Owner이며, 이동 대상 스레드와 옮겨갈 엔드포인트가 같은 프로젝트에 있음 |
| 기본 흐름 | 1. Owner가 옮길 최상위 댓글(스레드)에서 "댓글 이동" 체크박스를 활성화 2. 소프트 삭제되지 않은 엔드포인트 중 이동 대상을 선택 3. 시스템이 해당 최상위 댓글과 그에 달린 대댓글을 **묶음 단위로** 대상 엔드포인트로 옮김 4. 원래 위치에서는 사라지고, 대상 엔드포인트의 댓글 스레드에 나타남(작성자/시간/대댓글 구조는 그대로 유지) |
| 대안 흐름 | 체크박스를 활성화했더라도 대상을 선택하지 않고 취소하면 스레드는 원래 위치에 그대로 남음 |
| 예외 흐름 | 대상으로 **소프트 삭제된 엔드포인트**를 선택할 수 없음(선택지 자체에서 제외). 이동 대상은 현존(삭제되지 않은) 엔드포인트로 한정. **Member는 이동 기능 자체에 접근할 수 없음**(체크박스 미노출 또는 비활성) |
| 비고 | 이동 단위는 **최상위 댓글 + 하위 대댓글 전체**. 대댓글 하나만 분리 이동은 맥락 단절 때문에 허용하지 않음. 주 사용 맥락은 `path+method` 변경으로 엔드포인트가 소프트 삭제됐을 때, 살아있는 새 엔드포인트로 논의를 재배치하는 것. 이동 권한을 Owner로 한정한 이유는 스펙 구조 변경의 후속 정리라는 성격 때문(스펙 등록·업데이트 권한과 동일선상). 멘션·엔드포인트 멘션·리액션은 이동 후에도 그대로 유지. **UC-11과의 관계**: 이동은 Owner가 체크박스로 명시적으로 트리거하는 별도 액션이며 자동 실행되지 않음 — 이동을 실행하기 전까지 소프트 삭제된 엔드포인트의 댓글 스레드는 UC-11대로 원래 자리에 그대로 노출·열람된다 |
| 사용 예시 | `POST /courses`가 `POST /courses/create`로 경로가 바뀌어 기존 `POST /courses`가 소프트 삭제되자, Owner가 거기 쌓여 있던 "credits 기본값 논의" 스레드를 살아있는 `POST /courses/create`로 이동 → 논의 맥락이 새 엔드포인트에서 이어짐 |

### UC-13. 엔드포인트 댓글 스레드 AI 요약

| 항목 | 내용 |
| --- | --- |
| 액터 | Owner, Member, Agent |
| 목표 | 선택한 엔드포인트에 지금까지 쌓인 댓글·대댓글을 AI가 분석해, 논의 흐름을 요약한 댓글을 스레드에 남긴다 |
| 사전조건 | 프로젝트에 진입해 대상 엔드포인트를 선택한 상태이며, 해당 엔드포인트에 요약할 댓글이 1개 이상 존재 |
| 기본 흐름 | 1. 사용자가 해당 엔드포인트에서 "AI 요약" 실행 2. 시스템이 그 엔드포인트의 댓글·대댓글 전체(삭제되지 않은 내용)를 수집해 분석 3. 분석 결과를 요약 텍스트로 생성 4. 그 요약을 **전역 AI 계정을 작성자로 하는 새 최상위 댓글**로 해당 엔드포인트 스레드에 등록 5. 작성자(AI 계정)/작성 시간과 함께 댓글 목록에 표시 |
| 예외 흐름 | 1. 요약할 댓글이 없으면 요약 실행 불가 안내 2. AI 생성 실패 시 에러 안내, 댓글은 생성되지 않음 |
| 비고 | 요약 대상 범위는 **선택된 단일 엔드포인트의 스레드로 한정**(다른 엔드포인트·태그 논의는 포함하지 않음). 작성자는 각 프로젝트 멤버가 아니라 **서비스가 사전 생성해 둔 전역 AI 계정**이며, 초대·멤버십·멘션 대상에서는 제외됨. 생성된 AI 요약 댓글은 구조상 일반 최상위 댓글과 동일하게 취급되어 대댓글·리액션이 달릴 수 있으나, 사람 사용자의 "본인 글 수정/삭제"(UC-5) 대상은 아님. 요약은 실행 시점 스냅샷이며 이후 댓글이 추가돼도 자동 갱신되지 않음(다시 요약하면 새 요약 댓글이 별도로 생성됨). AI 요약 댓글도 UC-12(스레드 이동) 대상이 될 수 있음 |
| 사용 예시 | `POST /courses`에 credits 기본값 논의, 정원 처리 방식 등 댓글·대댓글이 십수 개 쌓이자, 신규 합류한 프론트엔드 개발자가 "AI 요약"을 실행 → AI 계정 작성자로 "이 엔드포인트 논의 요약: credits 기본값은 3으로 합의, 정원 초과 시 409 응답으로 결정됨" 같은 요약 댓글이 스레드에 생성됨 |
