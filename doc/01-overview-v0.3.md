## **1. 문제 인식**

Swagger UI는 OpenAPI 스펙을 사람이 읽을 수 있는 형태로 렌더링하는 표준 도구지만,
두 가지 구조적 한계가 있음.

1. **탐색 구조의 한계**: API 수가 늘어나면 원하는 엔드포인트를 찾기 위해 스크롤과 펼치기를 반복해야 함.
2. **협업 기능의 부재**: API 설계에 대한 논의가 Slack 등 외부 채널에 흩어져서 시간이 지나면 검색이 어려움.

## **2. 시장 상황**

| 도구 | 핵심 기능 |
| --- | --- |
| **Scalar** | 모던 UI 렌더링, 코드 샘플 언어별 자동 생성, 빠른 탐색. Swagger에서 마이그레이션 간단. 현재 .NET 9/10 기본 렌더러로 채택됨 |
| **Redoc / Redocly** | 3단 레이아웃의 읽기 중심 문서화 |
| **Stoplight** | 스펙 작성용 비주얼 에디터, 스타일 가이드 린팅(Spectral), 목 서버 제공 |
| **apiDog** | 디자인·테스트·모킹·문서화 통합 플랫폼, 엔드포인트 단위 멘션 기능, SaaS 방식 |

→ 렌더링·모던 UI·탐색 효율은 Scalar가 이미 무료·오픈소스로 해결함. 이 영역은 차별점이 될 수 없음.

→ 개발 과정에서 여러 부서와 끊임없이 소통해야 하는데, 그 기록은 엑셀이나 슬랙에 흩어짐.

→ 수요 근거 1: 개발자 구두 설문에서 필요성 공감.

→ 수요 근거 2: apiDog의 협업 기능 존재 자체가 수요의 방증.

→ 차별점 : 스펙 옆에 논의가 붙어서 검색 가능하게 쌓이는 기능 (apiDog만 이 영역을 점유하나 무거움)

→ 결론

> Swagger 스펙을 모던하게 보여주되, 각 엔드포인트 별로 팀원들이 댓글을 쉽게 남기고 쌓아둘 수 있도록 하자. 스펙 문서와 논의 기록을 한 곳에 문서화하자.
>

1. OpenAPI 스펙을 재구성하여 탐색 효율 상승
2. 엔드포인트별 댓글 스레드를 통해 협업을 지원
3. 스펙 문서와 논의 기록을 한곳에서 문서화
4. 사용이 쉽고 단순할 것 (주객전도 방지)

## 8. 접근 제어 및 역할 모델

- SpecNote 안에 여러 사용자와 여러 프로젝트가 공존
- 사용자 ↔ 프로젝트는 다대다 관계
- 역할은 계정 전역이 아니라 프로젝트마다 상대적으로 결정됨
    - 프로젝트를 생성한 사람 = Owner
    - 초대받아 들어온 사람 = Member
    - 같은 사용자가 A 프로젝트에선 Owner, B 프로젝트에선 Member일 수 있음
- 초대되지 않은 사용자는 해당 프로젝트에 접근 불가 (스펙 조회 · 댓글 스레드 전부 차단)
- 전역 관리자 개념 없음. 누구나 가입 후 프로젝트를 생성할 수 있음

| 동작 | Owner | Member |
| --- | --- | --- |
| 프로젝트 생성 | (생성자가 곧 Owner) | — |
| 프로젝트 수정/삭제 · 스펙 입력 · 스펙 업데이트 | O | X |
| 멤버 초대 / 제외 | O | X |
| 엔드포인트 조회 | O | O |
| 댓글 스레드 작성·수정·삭제(본인 것) | O | O |

## 9. 가입 및 초대 방식

- **가입**: 이메일 + 이름 + 비밀번호. 비밀번호는 해시 저장(bcrypt 등)
    - 이메일 인증(verification) 없음 (Phase3에서 논의)
    - 비밀번호 찾기 없음 (Phase3에서 논의)
- 초대: Owner가 이메일 완전일치로 기존 가입자를 검색하여 초대
    - 초대 대상은 사전에 서비스에 가입되어 있어야 함
    - 이름은 검색 키가 아니라 표시용 (전체 유저 명단 노출 방지)
