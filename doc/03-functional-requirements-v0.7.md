## FR-1. 프로젝트 생성 및 스펙 등록 (관련 UC-1)

| ID     | 요구사항                                                                                                                                                                                                                                                                                                                        | 우선순위 |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| FR-1.1 | 로그인한 사용자는 새 프로젝트를 생성할 수 있어야 하며, 생성자는 해당 프로젝트의 Owner 역할을 자동으로 부여받아야 한다                                                                                                                                                                                                           | P1       |
| FR-1.2 | Owner는 프로젝트 생성 시 OpenAPI 스펙을 받아올 주소(URL)를 입력해야 하며, 시스템은 해당 주소에서 스펙을 가져와 파싱해야 한다                                                                                                                                                                                                    | P1       |
| FR-1.3 | 시스템은 파싱한 엔드포인트(path, method, summary 등)를 해당 프로젝트 스코프로 구분하여 DB에 저장해야 한다                                                                                                                                                                                                                       | P1       |
| FR-1.4 | 시스템은 입력된 원본 스펙 JSON 전체를 그 프로젝트의 첫 번째(v1) 버전 스냅샷으로 저장해야 하며, 스냅샷은 덮어쓰지 않고 append-only로 누적해야 한다                                                                                                                                                                               | P1       |
| FR-1.5 | 시스템은 입력 내용이 OpenAPI 형식이 아니거나 주소 접근·파싱에 실패하면 명확한 에러 메시지를 표시하고 저장을 취소해야 한다                                                                                                                                                                                                       | P1       |
| FR-1.6 | Owner는 Try it out에 사용할 API 서버 base URL을 프로젝트 생성 시 입력할 수 있어야 하며(비필수), 이 값은 스펙 JSON 소스와 독립적이어야 한다                                                                                                                                                                                      | P1       |
| FR-1.7 | Owner는 프로젝트의 등록된 스펙 URL과 Try it out base URL 등 사용자 설정 값을 수정할 수 있어야 하며, 이 권한은 Owner에 한정되어야 한다. 단, 스펙 파싱값(`title`/`description`/`version`/`oasVersion`)은 직접 수정 대상이 아니다                                                                                                  | P1       |
| FR-1.8 | Owner는 프로젝트를 삭제할 수 있어야 하며, 삭제 시 해당 프로젝트의 스펙·엔드포인트·댓글 등 데이터는 유지되어야 한다.(소프트 딜리트) 이 권한은 Owner에 한정되어야 한다. 단, 삭제된 프로젝트는 모든 멤버(Owner 포함)의 접근이 차단되어야 한다                                                                                      | P1       |
| FR-1.9 | 시스템은 프로젝트의 `title`/`description`/`version`/`oasVersion`을 사용자 입력이 아니라 스펙의 `info.*`·`openapi` 필드 파싱값으로 저장해야 하며, `description`을 제외한 나머지는 필수로 스펙 로드 시 자동 생성되어야 한다. Owner는 이 값들을 텍스트로 직접 수정할 수 없고, 스펙 리로드(FR-10) 시에만 최신값으로 갱신되어야 한다 | P1       |

## FR-2. 멤버 초대·제외 및 접근 제어 (관련 UC-2, 개요 #8)

| ID     | 요구사항                                                                                                                                                 | 우선순위 |
| ------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| FR-2.1 | Owner는 이메일 완전일치로 기존 가입자를 검색하여 프로젝트의 Member로 초대할 수 있어야 하며, 초대 대상은 사전에 서비스에 가입되어 있어야 한다             | P1       |
| FR-2.2 | 시스템은 이름을 검색 키로 사용하지 않고 표시용으로만 사용하여 전체 유저 명단이 노출되지 않도록 해야 한다                                                 | P1       |
| FR-2.3 | 시스템은 멤버 초대 시 초대 대상에게 서비스 내부 알림을 발송해야 한다 (메일 발송은 범위 밖)                                                               | P1       |
| FR-2.4 | Owner는 멤버를 프로젝트에서 제외할 수 있어야 하며, 제외된 사용자는 즉시 해당 프로젝트 접근이 차단되어야 한다                                             | P1       |
| FR-2.5 | 시스템은 멤버십을 물리 삭제하지 않고 `isDeleted` 단일 boolean 소프트 딜리트 마커로 제외 여부만 표시해야 하며, 별도 이력 로그는 두지 않는다               | P1       |
| FR-2.6 | 시스템은 초대되지 않은 사용자의 프로젝트 스펙 조회·댓글 열람을 전면 차단해야 하며, 모든 데이터 접근은 프로젝트 스코프로 제한되어야 한다                  | P1       |
| FR-2.7 | 시스템은 역할을 계정 전역이 아니라 프로젝트마다 상대적으로 부여해야 하며, 같은 사용자가 A 프로젝트에서는 Owner, B 프로젝트에서는 Member일 수 있어야 한다 | P1       |
| FR-2.8 | 이메일이 가입자와 완전일치하지 않으면 시스템은 초대 대상 없음으로 처리해야 한다                                                                          | P1       |

## FR-3. 프로젝트 선택 및 스펙 탐색 (관련 UC-3)

| ID     | 요구사항                                                                                                                                       | 우선순위 |
| ------ | ---------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| FR-3.1 | 시스템은 프로젝트 목록에 사용자가 Owner인 프로젝트와 Member로 초대된 프로젝트만 노출해야 한다                                                  | P1       |
| FR-3.2 | 사용자는 접근 권한이 있는 프로젝트를 선택해 진입할 수 있어야 하며, 진입 후 좌측 사이드바에 태그별로 그룹핑된 엔드포인트 목록이 표시되어야 한다 | P1       |
| FR-3.3 | 시스템은 사이드바에서 엔드포인트를 클릭하면 우측 디테일 패널을 해당 엔드포인트 정보로 갱신해야 한다                                            | P1       |
| FR-3.4 | 시스템은 검색어 입력 시 엔드포인트 목록을 실시간으로 필터링해야 한다                                                                           | P1       |

## FR-4. 엔드포인트 상세 확인 및 요청 테스트 (관련 UC-4)

| ID     | 요구사항                                                                                                                                                                                                                                          | 우선순위 |
| ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- |
| FR-4.1 | 디테일 패널은 선택된 엔드포인트의 요청 스키마(필드명·타입)를 표시해야 한다                                                                                                                                                                        | P1       |
| FR-4.2 | 디테일 패널은 응답 스키마와 example 값을 표시해야 한다                                                                                                                                                                                            | P1       |
| FR-4.3 | 시스템은 배열 안에 중첩된 객체 구조(예: `timeTables`의 `dayOfWeek`/`hour`)를 재귀적으로 펼쳐 표시할 수 있어야 한다                                                                                                                                | P1       |
| FR-4.4 | 사용자는 "Try it out"으로 example 값 기반의 요청 파라미터/바디를 입력(수정 가능)하여 프로젝트에 등록된 base URL로 실제 요청을 전송하고, 실제 응답(상태 코드·바디)을 화면에서 확인할 수 있어야 한다                                                | P1       |
| FR-4.5 | 시스템은 프로젝트에 base URL이 입력되지 않았으면 Try it out을 비활성화해야 하며, 이때에도 스펙 열람·댓글 등 나머지 기능은 정상 동작해야 한다                                                                                                      | P1       |
| FR-4.6 | 시스템은 대상 API가 SpecNote 오리진의 요청을 CORS로 허용하지 않아 요청이 실패하면 원인이 대상 API의 CORS 설정에 있음을 사용자에게 안내해야 한다 (브라우저→대상 API 직접 요청 구조, 목 서버 없음)                                                  | P1       |
| FR-4.7 | Owner는 프로젝트 설정에서 base URL을 수정할 수 있어야 하며(스펙 등록·업데이트와 동일한 권한선상), Member는 수정할 수 없어야 한다                                                                                                                  | P1       |
| FR-4.8 | 시스템은 Try it out에서 인증이 필요한 요청을 위해 전역 Bearer 토큰 입력을 제공해야 하며, 입력된 토큰을 요청 헤더에 포함해 전송해야 한다. 지원 범위는 http bearer 타입에 한정하며, oauth2 등 그 외 인증 타입은 미지원임을 사용자에게 안내해야 한다 | P1       |

## FR-7. 댓글/대댓글 팀원 멘션 (관련 UC-7)
